// Foundation
Programming Ki History — OOP Kyun Aaya?
Pehle samjho kahan se aaye — tabhi OOP ki asli zaroorat samjhoge
Era 1

Machine Language — 0101...

Seedha binary mein CPU se baat. Ek bit galat — pura code khatam. Error-prone, tedious, zero scalability. Large app banana impossible.

Era 2

Assembly Language — MOV A, 61H

English keywords (mnemonics) aaye. Par hardware se tight coupling thi — hardware badla to code badlna padta. No proper loops/functions. Still impractical.

Era 3

Procedural — C Language

Functions, loops, if/else sab aaya. Code "recipe book" ki tarah tha. Problem: real-world model nahi karta, data secure nahi, large-scale = spaghetti code.

Era 4

Object Oriented — C++, Java, Python, TypeScript...

Real-world objects ko code mein laao. Data secure karo. Code reuse karo. Aaj ke sabhi enterprise apps isi pe based hain.

⚠️ Procedural ki 3 Badi Problems

  • Real-world model nahi hota — duniya mein objects hain, sirf functions nahi
  • Data Security nahi — koi bhi variable directly change kar sakta tha
  • Large apps mein "Spaghetti Code" — maintain karna nightmare

✅ OOP ne solve kiya

  • Objects se real-world entities represent karo — Uber, Zomato build karna easy
  • Private members se data protect karo (Encapsulation)
  • Inheritance se code reuse, Polymorphism se flexible code

Class & Object
OOP ki foundation — class = blueprint, object = usse bani cheez
Class
[ Blueprint / Template / Mold ]
Class ek blueprint hai. Jaise architect ka ghar ka design — usme rooms ka plan hai par woh ghar nahi. Class mein define karte hain ki object ki kya properties (data) aur kya methods (behaviors) honge. Memory tab allocate hoti hai jab new se object banta hai.
Object
[ Instance — Real entity in memory ]
Object ek actual instance hai class ka. Har object apni alag state rakhta hai. Ek class se kitne bhi objects ban sakte hain. new ClassName() se banta hai. Methods class ke prototype mein shared hote hain — data alag-alag.
🏠
Perfect Analogy: Ek house blueprint (class) se kai actual houses (objects) ban sakte hain. Blueprint mein plan hai — bedrooms, doors, windows. Har actual ghar ka apna address hai, apna color hai — par sab ka blueprint ek hi. Isi tarah Car class ek blueprint hai, myCar aur yourCar alag objects hain.

class Car {
  // Instance Properties — har object ki apni copy hoti hai
  brand:        string;
  model:        string;
  color:        string;
  currentSpeed: number;
  isEngineOn:   boolean;

  constructor(brand: string, model: string, color: string = "White") {
    this.brand        = brand;
    this.model        = model;
    this.color        = color;
    this.currentSpeed = 0;
    this.isEngineOn   = false;
  }

  // Methods — object ke behaviors
  startEngine(): void {
    this.isEngineOn = true;
    console.log(`🚗 ${this.brand} ${this.model} started!`);
  }

  accelerate(speed: number): void {
    if (!this.isEngineOn) { console.log("Start engine first!"); return; }
    this.currentSpeed += speed;
    console.log(`Speed: ${this.currentSpeed} km/h`);
  }

  brake(): void {
    this.currentSpeed = Math.max(0, this.currentSpeed - 30);
  }

  stopEngine(): void {
    this.currentSpeed = 0;
    this.isEngineOn   = false;
  }

  toString(): string {
    return `Car(${this.brand}, ${this.model}, ${this.currentSpeed}km/h)`;
  }
}

Access Modifiers
Kisko kya dikhega aur access milega — yeh control karte hain

🟢 Public

Sabko access. Class ke bahar, andar, child class — sab jagah. TypeScript mein default hi public hai.

🔴 Private

Sirf usi class ke andar. Child class bhi access nahi kar sakti. Sensitive data protect karne ke liye. Getters/Setters se controlled access dete hain.

🟡 Protected

Class + Child Classes. Bahar se nahi — par inheritance ke liye accessible. Parent ka data child use kare — protected use karo.

// TypeScript — All Access Modifiers
class BankAccount {
  public    accountNumber: string;   // Har jagah accessible
  protected balance:       number;   // Class + child classes
  private   pin:           string;   // Sirf is class mein
  readonly  createdAt:     Date;     // Ek baar set, change nahi

  // TypeScript private shorthand (#) — JS native private
  // Class fields proposal se aaya — ACTUAL runtime enforcement!
  #secretKey: string;  // private keyword se alag — runtime bhi block karta

  constructor(accNum: string, pin: string, balance: number) {
    this.accountNumber = accNum;
    this.pin           = pin;
    this.balance       = balance;
    this.createdAt     = new Date();
    this.#secretKey    = "SK_" + Math.random();
  }

  public getBalance(): number { return this.balance; }

  public verifyPin(entered: string): boolean {
    return this.pin === entered; // Inside class — private OK hai
  }

  public deposit(amount: number): void {
    if (amount <= 0) throw new Error("Invalid amount");
    this.balance += amount;
  }
}

Static

// TypeScript — Static Members Complete
class Car {
  // ─── Static Properties ─────────────────────────
  static        totalCars: number = 0;            // Shared counter
  static readonly MAX_SPEED: number = 300;       // Constant
  private static allCars: Car[] = [];            // Registry

  // Instance properties
  brand: string;
  model: string;
  readonly id: number;

  constructor(brand: string, model: string) {
    this.brand = brand;
    this.model = model;
    Car.totalCars++;                        // Class name se access
    this.id = Car.totalCars;               // Unique ID assign
    Car.allCars.push(this);
  }

  // ─── Static Methods ────────────────────────────
  static getTotal(): number {
    return Car.totalCars;
  }

  static getAllCars(): Car[] {
    return [...Car.allCars]; // copy return karo
  }

  // Static factory method — alternative constructor!
  static fromString(str: string): Car {
    const [brand, model] = str.split("-");
    return new Car(brand, model);
  }

  // Instance method
  getInfo(): string {
    return `#${this.id}: ${this.brand} ${this.model}`;
  }
}

Getters & Setters
Private data ko controlled read/write access — validation ke saath

Getter (get)

Private property ki value padhne ke liye. Property ki tarah access hota hai — parentheses nahi lagti. Computed properties bhi return kar sakte hain. obj.speed — looks like property, actually method hai.

Setter (set)

Private property ki value likhne ke liye — par validation ke saath! Seedha obj._speed = 500 se nahi, setter se hoga — toh pehle validate kar sakte hain. Isliye private + getter/setter prefer karo over public property.

// TypeScript — get/set syntax
class SportsCar {
  private _brand:  string;
  private _speed:  number;
  private _tyre:   string;
  private _fuel:   number;

  constructor(brand: string) {
    this._brand = brand;
    this._speed = 0;
    this._tyre  = "MRF";
    this._fuel  = 100;
  }

  // ─── GETTER ONLY (read-only property) ─────────────
  get speed(): number { return this._speed; }
  get brand(): string { return this._brand; }

  // Computed getter — data se calculate karta hai
  get isMoving(): boolean { return this._speed > 0; }
  get speedKmph(): string  { return `${this._speed} km/h`; }

  // ─── GETTER + SETTER (with validation!) ───────────
  get tyre(): string { return this._tyre; }

  set tyre(name: string) {
    const valid = ["MRF", "Bridgestone", "Michelin", "Apollo"];
    if (!name?.trim()) throw new Error("Tyre name empty!");
    if (!valid.includes(name)) throw new Error(`Invalid: ${name}`);
    this._tyre = name;
  }

  get fuel(): number { return this._fuel; }
  set fuel(val: number) {
    if (val < 0 || val > 100) throw new Error("Fuel 0-100 range!");
    this._fuel = val;
  }

  accelerate(s: number) { this._speed += s; }
}

Sabse Common Confusion — Abstraction vs Encapsulation

Abstraction — "Complexity hide karo." User ko sirf interface dikhao — andar kya ho raha hai ushe mat dikhao. Design level decision.

Encapsulation — "Data secure karo." Private members se unauthorized access rokh. Getters/setters se controlled access. Implementation level decision.

Simple yaad karo: Abstraction = WHAT (kya kar sakta hai) | Encapsulation = HOW PROTECTED (data kitna safe hai)

    if (amount > this._balance)       { console.log("Insufficient!"); return false; }
    if (amount <= 0)                  { console.log("Invalid!"); return false; }

    this.logTransaction(`Withdraw: -₹${amount}`);
    return true;

Overloading: Same class, same name, alag parameters. Compile time. add(int,int) vs add(double,double).

Overriding: Parent-child, same name, same parameters — par child ki apni implementation. Runtime. Circle.area() overrides Shape.area().

Hiding (C++): virtual keyword nahi lagaya to child parent ka method hide karta hai — override nahi. Virtual dispatch nahi hota. Mostly avoid karo!

// ─── Abstract class — Database connector example ─────
abstract class DatabaseConnector {
  protected isConnected: boolean  = false;
  protected queryCount: number   = 0;

  constructor(protected readonly host: string, protected readonly port: number) {}

  // ─── CONCRETE methods — common logic ─────────────
  connect(): void {
    this.isConnected = true;
    console.log(`Connected to ${this.host}:${this.port}`);
  }

  disconnect(): void {
    this.isConnected = false;
    console.log(`Disconnected. Total queries: ${this.queryCount}`);
  }

  executeQuery(sql: string): any {
    if (!this.isConnected) throw new Error("Not connected!");
    this.queryCount++;
    return this.runQuery(sql);  // delegates to abstract method
  }

  getStats(): string {
    return `${this.getDbType()} @ ${this.host} | ${this.queryCount} queries`;
  }

  // ─── ABSTRACT methods — child must implement ─────
  abstract getDbType(): string;
  protected abstract runQuery(sql: string): any;
  abstract createTable(name: string, schema: object): void;
}

class PostgresConnector extends DatabaseConnector {
  constructor(host: string) { super(host, 5432); }

  getDbType(): string { return "PostgreSQL"; }

  protected runQuery(sql: string): any {
    console.log(`[Postgres] Executing: ${sql}`);
    return { rows: [], rowCount: 0 };
  }

  createTable(name: string, schema: object): void {
    console.log(`[Postgres] CREATE TABLE ${name}...`);
  }

  // Postgres-specific method
  vacuum(): void { console.log("[Postgres] VACUUM running..."); }
}

class DatabaseConnector(ABC):
    def __init__(self, host: str, port: int):
        self._host        = host
        self._port        = port
        self._connected   = False
        self._query_count = 0

    # Concrete
    def connect(self):
        self._connected = True
        print(f"Connected to {self._host}:{self._port}")

    def disconnect(self):
        self._connected = False
        print(f"Disconnected. Queries: {self._query_count}")

    def execute_query(self, sql: str):
        if not self._connected: raise RuntimeError("Not connected!")
        self._query_count += 1
        return self._run_query(sql)

    @abstractmethod
    def get_db_type(self) -> str: ...

    @abstractmethod
    def _run_query(self, sql: str) -> Any: ...

abstract class BaseEntity implements ISerializable, IAuditable {

    this.createdAt = new Date();

  abstract getDisplayName(): string;
  abstract getSummary(): string;
}

// ─── GENERIC RESULT TYPE (error handling pattern) ────
class Result<T, E = Error> {
  private constructor(
    private readonly _value?: T,
    private readonly _error?: E
  ) {}

  static ok<T>(value: T):     Result<T> { return new Result<T>(value); }
  static fail<T>(err: Error): Result<T> { return new Result<T>(undefined, err); }

  isOk():  boolean     { return this._error === undefined; }
  unwrap(): T            { if (this._error) throw this._error; return this._value!; }
  unwrapOr(fallback: T): T { return this.isOk() ? this._value! : fallback; }
  getError(): E | undefined { return this._error; }
}

function divide(a: number, b: number): Result<number> {
  if (b === 0) return Result.fail(new Error("Division by zero!"));
  return Result.ok(a / b);
}

class QueryBuilder {
  private _table:      string   = "";
  private _conditions: string[] = [];
  private _columns:    string[] = [];
  private _orderBy:    string   = "";
  private _limit:      number   = -1;
  private _offset:     number   = 0;
  private _joins:      string[] = [];

  // Each method returns 'this' — enables chaining!
  from(table: string): this {
    this._table = table;
    return this;
  }

  select(...cols: string[]): this {
    this._columns.push(...cols);
    return this;
  }

  where(condition: string): this {
    this._conditions.push(condition);
    return this;
  }

  join(table: string, on: string): this {
    this._joins.push(`JOIN ${table} ON ${on}`);
    return this;
  }

  orderBy(col: string, dir: "ASC" | "DESC" = "ASC"): this {
    this._orderBy = `${col} ${dir}`;
    return this;
  }

  limit(n: number): this  { this._limit  = n; return this; }
  offset(n: number): this { this._offset = n; return this; }

  build(): string {
    const cols = this._columns.length ? this._columns.join(", ") : "*";
    let sql = `SELECT ${cols} FROM ${this._table}`;
    if (this._joins.length)     sql += ` ${this._joins.join(" ")}`;
    if (this._conditions.length) sql += ` WHERE ${this._conditions.join(" AND ")}`;
    if (this._orderBy)           sql += ` ORDER BY ${this._orderBy}`;
    if (this._limit > 0)        sql += ` LIMIT ${this._limit}`;
    if (this._offset > 0)       sql += ` OFFSET ${this._offset}`;
    return sql;
  }
}

function Memoize(target: any, key: string, desc: PropertyDescriptor) {
  const original = desc.value;
  const cache    = new Map<string, any>();

  desc.value = function(...args: any[]) {
    const cacheKey = JSON.stringify(args);
    if (cache.has(cacheKey)) {
      console.log(`[Cache hit] ${key}(${cacheKey})`);
      return cache.get(cacheKey);
    }
    const result = original.apply(this, args);
    cache.set(cacheKey, result);
    return result;
  };
}

function Validate(target: any, key: string, desc: PropertyDescriptor) {
  const orig = desc.value;

    for (const arg of args) {
      if (arg === null || arg === undefined)
        throw new Error(`${key}: null/undefined not allowed!`);
    }
    return orig.apply(this, args);
  };
}

  abstract getEntityName(): string;
}

interface IUser    extends IEntity { name: string; email: string; }
interface IProduct extends IEntity { name: string; price: number; }

class UserRepo extends BaseRepository<IUser> {
  validate(user: IUser): boolean {
    return !!user.name && user.email.includes("@");
  }
  getEntityName(): string { return "User"; }

  // User-specific method
  findByEmail(email: string): IUser | undefined {
    return this.items.find(u => u.email === email);
  }
}

class ProductRepo extends BaseRepository<IProduct> {
  validate(p: IProduct): boolean { return !!p.name && p.price >= 0; }
  getEntityName(): string         { return "Product"; }

  findUnder(price: number): IProduct[] {
    return this.items.filter(p => p.price < price);
  }
}
